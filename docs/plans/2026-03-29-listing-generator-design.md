# Zolan Listing Generator — Design Document

**Date:** 2026-03-29
**Status:** Approved

## Overview

A progressive web app (PWA) that generates e-commerce product listings for multiple platforms simultaneously. Users describe their product (text + images + PDFs), select target platforms, and receive tailored markdown listings per platform. Solves the context-loss problem of using ChatGPT for repeated listing generation by explicitly managing the full context window on every API call.

### Target Users

Small-group of resellers (friends) selling on eBay, Poshmark, Depop, Mercari, etc. Primary use on desktop, heavy secondary use on mobile. Multiple users, each with their own account and API key.

## Agent Pipeline

### Step 1: Classification
- **Model**: GPT-4o-mini (cheap, fast)
- **Input**: user prompt + image descriptions
- **Output** (structured JSON):
  - needs_web_search: boolean
  - search_queries: string[] (if needed)
  - product_category: string
  - detected_brand: string | null
  - detected_attributes: object

### Step 2: Enrichment (conditional, parallel)
- **Web Search** (if classification says yes): Tavily search per query, returns pricing data and product details
- **Image Analysis** (if images uploaded): vision model describes product from photos
- These run in parallel

### Step 3: Generation (parallel per platform)
- **Model**: user's selected model
- For EACH selected platform simultaneously, system message composed from:
  1. Platform-specific instructions (eBay: 80 char title, bullet points; Poshmark: 20 word stylish title, etc.)
  2. User's custom system prompt (if set)
  3. PDF extracted text (if uploaded)
  4. Enrichment results (search + image analysis)
- User message: original prompt
- Images: attached for vision models
- Output: streaming markdown per platform

### Step 4: Title Generation
- **Model**: GPT-4o-mini
- Runs once per session (on first generation)
- Input: user's prompt (first ~200 chars)
- Output: short session title (5-8 words)

### Tool Definitions (available to main generation model)

```typescript
tools = [
  {
    type: "function",
    function: {
      name: "web_search",
      description: "Search the web for product pricing, recent sales data, or product information",
      parameters: {
        type: "object",
        properties: {
          query: { type: "string", description: "Search query" }
        },
        required: ["query"]
      }
    }
  }
]
```

Classification handles most routing, but the main model can also invoke web_search mid-generation if it needs more info. Server intercepts the tool call, hits Tavily, feeds results back.

### Streaming UX
- Each platform card appears immediately in "loading" state
- Markdown streams in token-by-token, rendered live
- Cards can be expanded while still streaming
- If one platform finishes before another, it's immediately interactive

## Context Management (Core Value Proposition)

This is why this app exists — context is **never lost** because we control exactly what goes to the model every time.

- **System prompt**: injected first, every call, never lost
- **PDF text**: stored on the message, re-injected on follow-ups
- **Platform instructions**: always present per-platform
- **Conversation history**: all prior messages in the session included
- If context grows too large: summarize older messages (not truncate) using a cheap model

ChatGPT's problem is opaque context management. We make it explicit and deterministic.

## Authentication & Security

- **Registration**: username + password -> bcrypt hash -> SQLite
- **Login**: verify bcrypt -> set HttpOnly secure cookie (session token)
- **Session**: UUID token, 30-day expiry, rolling renewal
- **API Key Storage**: AES-256-GCM encrypted in SQLite, server-side encryption key from env var
- **API Key Display**: masked in UI (sk-or-...****7f2a), never sent back in full
- **Route Protection**: all /app/* routes require valid session cookie, validated in hooks.server.ts
- **No API key or generation data accessible without auth**

## File Upload & Processing

### Images
- Accepted: JPEG, PNG, WebP, HEIC
- Max: 20MB per image, up to 5 per message
- Storage: /data/uploads/{user_id}/{session_id}/{filename}
- HEIC auto-converted to JPEG (sharp)
- Thumbnails generated for UI preview
- Sent to vision model as base64 data URLs
- Cleanup: configurable retention (default 90 days)

### PDFs
- Accepted: .pdf, max 10MB
- Text extracted via pdf-parse
- Extracted text stored in message record
- Original PDF stored in uploads dir
- Text-based PDFs only (no OCR needed)

### UX
- Drag-and-drop on desktop
- File picker buttons for images and PDF
- Upload previews with remove button
- Progress indicator for large files

## OpenRouter Integration Details

- **Base URL**: https://openrouter.ai/api/v1
- **Auth**: Bearer token (user's BYOK key)
- **Compatibility**: Full OpenAI SDK compatible — use openai npm package with custom baseURL
- **Streaming**: SSE with `stream: true`, content in `choices[0].delta.content`
- **Tool calling**: Standard OpenAI format, OpenRouter normalizes across providers
- **Key validation**: GET /api/v1/key returns credit balance and limits
- **Attribution headers**: HTTP-Referer and X-OpenRouter-Title
- **Keepalive**: Ignore `: OPENROUTER PROCESSING` SSE comments
- **Errors**: 402 (no credits), 429 (rate limit), standard HTTP otherwise
